## Problem Statement

The Seichijunrei agent has one eval layer: an end-to-end pipeline eval (`test_plan_quality.py`) that runs 163 cases through the full `run_pipeline()` path. It takes 1-2 hours, burns model quota, and when a case fails you can't tell whether the problem is in the intent classifier, the planner's LLM reasoning, the result_validator's guard logic, the executor's dispatch, or the database.

The system is fully ReAct-driven. Every request follows:
```
classify_intent() → planner.step() [LLM] → executor._execute_step() [deterministic] → loop
```
> [Verified 2026-04-11]: `classify_intent` in `backend/agents/intent_classifier.py`, `planner.step()` in `backend/agents/planner_agent.py`, `react_loop()` in `backend/agents/pipeline.py`, `ReactStep` model in `backend/agents/models.py`. All paths confirmed.

There is no one-shot plan path in production. The eval architecture should mirror this execution reality.

## What Makes This Cool

A proper eval pyramid lets you iterate on the agent in minutes instead of hours. Change a prompt? Run Layer 1b in 3 minutes. Change a validator rule? Run Layer 1a in 5 seconds. The slow pipeline eval becomes a CI-only safety net, not your daily feedback loop.

## Constraints

- Must reuse the existing 163-case dataset (`plan_quality_v1.json`)
- Must work with multi-provider fallback (Gemini 3.1 Pro, GPT 5.4 via Univibe)
- Must not break existing `make test-eval` workflow
- Eval layers must be independently runnable
- No new external dependencies (use pydantic_evals, pytest, existing stack)

## Premises

1. The 163-case dataset is reused across all eval layers (no separate datasets per layer)
2. Component evals split into deterministic (no LLM, seconds) and single-LLM (minutes)
3. The existing pipeline eval stays as Layer 3 but demotes to CI-only
4. DX improvements (model precheck, per-case progress, per-case timeout, first-run baseline+gate) ship as part of this work
5. Each layer produces its own baseline file and enforces its own gate independently
6. Any layer can target any configured model via EVAL_MODEL env var

## Approaches Considered

### Approach A: Monorepo eval module (CHOSEN)
All eval layers live in `backend/tests/eval/`, sharing dataset loader, baseline mechanism, and gate logic. Each layer is one file. Shared infrastructure is extracted to a common module.

Effort: M (~40 min CC). Risk: Low.
Pros: minimal diff, DRY shared infra, one `make` target per layer.
Cons: eval/ directory grows from 2 to ~8 files.

### Approach B: Eval as separate package
Create `backend/eval/` as independent module with CLI entry point.

Effort: L (~60 min CC). Risk: Medium.
Pros: conceptual separation, independent CI jobs.
Cons: more boilerplate, pytest integration glue, higher complexity.

### Approach C: Minimal viable (planner + ReAct only)
Only add `test_planner_quality.py` and `test_react_quality.py`.

Effort: S (~20 min CC). Risk: Low.
Pros: fastest to ship, fills biggest eval gap.
Cons: no deterministic component eval with full dataset, incomplete pyramid.

## Recommended Approach

**Approach A: Monorepo eval module.**

The complete eval pyramid, implemented as individual files under `backend/tests/eval/` with shared infrastructure.

### Execution model (current system)

  |     |-- planner.step()       <-- LLM call (structured output: ReactStep)
  |     |     |-- output_validator:    [Note: implemented as @agent.output_validator

## Layer Specifications

### Layer 1a: Component eval (deterministic, no LLM)

**File:** `test_component_quality.py`

**What it tests:**
- `classify_intent(query, locale)` → expected intent (from `backend/agents/intent_classifier.py`)
- `output_validator` behavior: given (history, intent, ReactStep), does it pass/reject correctly
  > [Note 2026-04-11]: The spec originally said "result_validator" but the actual implementation is an `@agent.output_validator` decorator in `planner_agent.py`. The test will need to extract or replicate the validator logic to test it independently.

**Entry point per case:**
```python
# Intent classifier
actual_intent, confidence = classify_intent(case.query, case.locale)
# Compare actual_intent to case.expected_intent

# Result validator (selected cases only)
# Build mock ReactStep + history, verify validator accepts/rejects
```

**Evaluators:**
- `IntentClassifierAccuracy`: score 1.0 if classify_intent matches expected
- `ValidatorBehavior`: For each case with `expected_steps`, construct two scenarios (tests `@agent.output_validator` from `planner_agent.py`):
  - **Positive**: build a history with all prerequisite steps completed, present a valid `done` ReactStep. Validator should accept.
  - **Negative**: build a history missing required prerequisites (e.g., no `search_bangumi` for ANIME_SEARCH intent), present a premature `done`. Validator should reject with `ModelRetry`.
  - Score 1.0 if both positive and negative pass. Target: all 163 cases produce at least one positive scenario; cases with 2+ expected steps produce a negative scenario too.

**Speed:** ~5 seconds for 163 cases. No LLM, no DB, no network.

**Dataset usage:** All 163 cases have `expected_intent` (verified: every case in the current dataset already has this field). For validator tests, `expected_steps` is used to construct mock history/ReactStep pairs.

**Baseline model ID:** Layer 1a uses `"deterministic"` as the sentinel model ID for baseline files, since results are model-independent.

### Layer 1b: Planner eval (single LLM call)

**File:** `test_planner_quality.py`

**What it tests:**
- Given a query, does `planner.step()` select the correct first tool?
- Is the thought coherent?

**Entry point per case:**
```python
planner = ReActPlannerAgent(model)
react_step = await planner.step(
    text=case.query,
    locale=case.locale,
    context=case.context,
    history=[],           # empty: first step
    classified_intent=classify_intent(case.query, case.locale)[0],
)
# Check react_step.action.tool matches case.expected_steps[0]
```

**Evaluators:**
- `FirstStepMatch`: score 1.0 if first action.tool == expected_steps[0]
- `ThoughtRelevance`: score 1.0 if `thought` is non-empty, longer than 10 characters, and does not match a blocklist of boilerplate phrases (e.g., "I will help you", "Let me think", "Sure", "OK"). Score 0.0 otherwise.

**Speed:** 163 cases x 1 LLM call = ~3-5 minutes assuming serial execution with typical ~1-2s latency per call. Worst case with per-case 60s timeout: ~163 minutes (unlikely unless provider is down, which precheck catches).

**Limitation:** This layer tests the planner's first-step decision in isolation with empty history. Multi-turn planner behavior is covered by Layer 2.

**Dataset usage:** Reads `expected_steps[0]` as the expected first tool. Multi-turn cases use `context` field.

### Layer 2: ReAct eval (multi-LLM calls)

**File:** `test_react_quality.py`

**What it tests:**
- Does the ReAct loop converge (reach `done`) within max_steps?
- Is the final step sequence correct?
- Does it recover from injected failures?
- Is the step count efficient (not too many retries)?

**Entry point per case:**
```python
planner = ReActPlannerAgent(model)
executor = ExecutorAgent(mock_db)

events = []
async for event in react_loop(
    text=case.query,
    planner=planner,
    executor=executor,
    locale=case.locale,
    context=case.context,
):
    events.append(event)

# Extract step sequence, check convergence, count steps
```

**Evaluators:**
- `StepsMatch`: score 1.0 if successful step tools == expected_steps (from current pipeline eval)
- `Convergence`: score 1.0 if loop ended with "done" event (not max_steps timeout or error)
- `Efficiency`: `score = min(1.0, len(expected_steps) / max(1, total_event_count))`. If total events <= expected + 1: score 1.0. If 2-3 extra: score 0.5. If 4+ extra: score 0.0.
- `FailureRecovery`: for `inject_failure` cases only, score 1.0 if the loop recovered and produced the expected step sequence despite the injected failure

**Speed:** 163 cases x ~3-5 LLM calls each = ~10-20 minutes assuming serial execution. Worst case with timeouts: several hours (unlikely with precheck).

**Dataset usage:** Full `expected_steps` sequence, `expected_intent`, `context`. Cases with `inject_failure` field trigger failure injection.

**Mock DB fidelity:** Layer 2 uses a fixture-based mock that returns realistic data keyed by `bangumi_id` from each dataset case. For search cases: mock returns 3-5 realistic point rows. For route cases: mock returns ordered points. For resolve cases: mock returns a valid `bangumi_id`. This ensures planner behavior matches production patterns (not empty-result divergence). The mock contract:
- `find_bangumi_by_title(title)` -> returns `bangumi_id` from dataset case or `"262243"` as default
- `query_bangumi_points(bangumi_id)` -> returns 3-5 fixture points with valid lat/lng
- `search_points_by_location(lat, lng, radius)` -> returns 3 nearby fixture points
- `query_nearby_points(lat, lng, radius)` -> same as above
- All other methods -> return empty/None (non-search paths don't need realistic data)

## DX Improvements

### 1. Model precheck (before any eval)
One lightweight API call before running 163 cases. Catches:
- Missing/invalid API key
- Exhausted quota
- Network/proxy issues
- Wrong base_url

### 2. Per-case progress
Use pydantic_evals callback or custom wrapper to show:
```
[  42/163] search-ja-07  ✓  (1.2s)
[  43/163] search-ja-08  ✗  (timeout)
```

### 3. Per-case timeout
`asyncio.timeout(60)` around each case evaluation. Prevents one stuck case from blocking the entire run.

### 4. First-run baseline + gate in one pass
Current: first run creates baseline (SKIP), second run enforces gate.
New: first run creates baseline AND reports scores. Second run enforces gate. No behavioral change, but clearer messaging.

### 5. Proxy-safe by default
`_clear_proxy_env()` and `trust_env=False` already implemented in `base.py`. Eval common reuses this.

## Resolved Questions

1. **Layer 2 DB:** Fixture-based mock with realistic data (resolved above in Layer 2 spec). Real DB via testcontainer stays in Layer 3 only. Rationale: Layer 2 isolates agent behavior from infra. Layer 3 catches retriever/DB bugs.
2. **Concurrency:** Start with `max_concurrency=1` for all LLM layers. Add `max_concurrency=3` as an opt-in flag after initial baselines are established. Rationale: avoid rate limit noise in baseline runs.
3. **Layer 1a in stable CI:** Yes. Add `test-eval-components` to `make test` since it has zero external dependencies. Rationale: deterministic component eval is as fast and reliable as unit tests.
