## Problem Statement

Anime fans who do 聖地巡礼 (seichijunrei) currently plan pilgrimages using 4+ disconnected tools: Anitabi for spot data, Google Maps for routing, blog posts for tips, spreadsheets for organization. No single tool provides intelligent route optimization. Existing apps (Seichi, JapanAnimeMaps, AnimeTrips) are databases with maps, not planners. The gap isn't more data, it's intelligence on top of existing data.

The builder is their own first user, a J-type planner who wants optimized routes days/weeks before a trip. Pain point: "I want to visit 12 spots from one anime without walking back and forth. I also want to discover nearby spots from other shows."

## What Makes This Cool

The competitive advantage is **temporal intelligence** — knowing not just WHERE spots are, but WHEN and HOW to visit them efficiently. Sun position for photography, transit constraints, dwell time estimation, last-train awareness. Nobody else in this space has an AI that understands the *rhythm* of a pilgrimage day.

The wedge is replacing "Anitabi → copy addresses → Google Maps → manual reordering" with one query: "響け！ユーフォニアム 宇治 半日" → optimized timed itinerary with one-click Google Maps export.

## Constraints

- Web-first (responsive). Native app is Phase 2 (photo overlay features).
- Light theme only (京吹夏季 palette). No dark mode.
- ExecutorAgent must remain deterministic (no LLM calls during execution).
- New UI components register in `frontend/components/generative/registry.ts` only.
- Backend must pass `make check` (lint + typecheck + test).
- No community/social features.
- New endpoints follow existing auth pattern (Cloudflare Worker enforces JWT/API key before reaching container).
- API cost ceiling: MVP uses Haversine distance heuristic (free). Google Maps Directions API is a Phase A.5 upgrade if heuristic proves insufficient. No paid API calls in MVP.

## Premises

1. **Data problem is solved.** Anitabi and Bangumi.tv already have thousands of spots. Build a smart layer on top of existing data sources via write-through retrieval.
2. **Route optimization is the wedge.** The first thing that makes someone switch from the current 4-tool workflow is an optimized, non-backtracking route.
3. **Web-first is enough for MVP.** The photo overlay/camera tool is the Ultimate vision. Route planner + nearby discovery ships as responsive web app first.
4. **Builder is representative user.** J-type planner pain points (plan ahead, no backtracking, efficiency) are shared by enough 巡礼 fans to build for.
5. **Agent API is a future play.** Ctrip/Agent integration is valuable but not MVP scope.

### Phase A Implementation (1-2 weeks)

**Backend changes:**
1. **Upgrade `_execute_plan_route` in place** (`backend/agents/executor_agent.py`):
   - Replace nearest-neighbor with **2-opt** heuristic (sufficient for <30 spots; Christofides is Phase B upgrade if quality proves insufficient)
   - Add `locked_order` support (user-pinned spots maintain position in optimized sequence)
   - Add `skip_ids` support (excluded spots removed before optimization)
   - Estimate transit time between stops using **Haversine distance + walking speed heuristic** (80m/min walking, no paid API in MVP)
   - Return `TimedItinerary` output (see schema below)
   - The planner agent continues to emit `plan_route` as the `ToolName` — no new tool name needed

2. **Add output models** in `backend/agents/models.py`:
   ```python
   class TimedStop(BaseModel):
       point_id: int
       name: str
       arrive: str          # "09:00" (HH:MM)
       depart: str          # "09:15"
       dwell_minutes: int   # based on pacing preset
       lat: float
       lng: float

   class TransitLeg(BaseModel):
       from_id: int
       to_id: int
       mode: Literal["walk"]  # Phase B widens to "walk" | "train" | "car"
       duration_minutes: int
       distance_m: float

   class TimedItinerary(BaseModel):
       stops: list[TimedStop]
       legs: list[TransitLeg]
       total_minutes: int
       total_distance_m: float
       spot_count: int
       pacing: str          # "chill" | "normal" | "packed"
   ```

3. **Dwell time estimation** (deterministic, no LLM):

   Dwell time is NOT a flat value per spot. It's computed from spot metadata:
   - **Base dwell** = `photo_count * base_per_photo` (each photo angle takes time to frame + shoot)
   - If `photo_count` is unknown or 0, use `default_base` minutes
   - Pacing preset applies a **multiplier** to the base:
     - **chill (ゆっくり):** 1.5x base + 1.2x transit buffer (extra time to enjoy the place)
     - **normal (普通):** 1.0x base + 1.0x transit buffer
     - **packed (詰め込み):** 0.6x base + 0.8x transit buffer (quick photo, move on)
   - `base_per_photo` = 3 minutes (frame + shoot + compare with anime screenshot)
   - `default_base` = 8 minutes (when photo count unknown)
   - Example: a spot with 5 photo angles → base 15min → chill 23min, normal 15min, packed 9min
   - Data source: `points` table already stores multiple screenshots per spot. `photo_count` = number of screenshot records for that `point_id`

4. **Google Maps export** (computed in executor, no new endpoint needed):
   - Generate Google Maps URL with ordered waypoints (max 10 waypoints per URL, split if needed)
   - Generate `.ics` calendar file content with stops as timed events
   - Both included in `TimedItinerary` response as `export_google_maps_url` and `export_ics` fields

**Frontend changes:**
5. **New `RoutePlannerWizard.tsx` component** (register in `registry.ts`):
   - Spot list with lock/skip/drag-reorder + anime screenshot thumbnails
   - Pacing selector (ゆっくり/普通/詰め込み)
   - Transport mode toggle (徒歩/電車/車)
   - Timed timeline sidebar
   - Export buttons (Google Maps link, calendar download)
   - Fits in existing Result Panel (right column) — chat stays in center column

6. **Extend `RouteVisualization.tsx`:**
   - Add transit time labels on route legs
   - Add dwell time display at each stop
   - Add summary stats (total time, distance, spot count)

## Failure Modes & Degradation

| Failure | Degradation | User experience |
|---|---|---|
| Anitabi API down | Use cached `points` table data (write-through ensures DB has prior results) | Transparent — user sees cached spots |
| Zero spots found | Return message "この作品の巡礼ポイントが見つかりませんでした" + suggest searching by location | Clear empty state |
| <2 spots for optimization | Skip 2-opt, return single-stop itinerary | Still useful |
| Google Maps URL exceeds 10 waypoints | Split into multiple URLs with "Part 1/2" labels | Graceful split |
| Haversine estimate wildly wrong (e.g., mountain terrain) | Phase A.5 upgrade to Google Maps Directions API for known problem areas | MVP accepts this limitation |

## Resolved Decisions

1. **Transit data:** MVP ships with **Haversine distance heuristic** (walking speed 80m/min). Google Maps Directions API is Phase A.5 upgrade if heuristic proves insufficient on real trips.
2. **Dwell time:** Computed from **photo_count per spot * base_per_photo**, then scaled by pacing multiplier. Not a flat value — spots with more photo angles get more time.
3. **Algorithm:** **2-opt** for MVP (simple, sufficient for <30 spots). Christofides or OR-Tools is Phase B.
4. **Tool naming:** Upgrade existing `plan_route` in place. No `plan_route_v2`. Planner continues emitting `plan_route`.
5. **Backward compatibility:** `plan_route` response shape changes from `{ordered_points, point_count, status, summary}` to include `TimedItinerary` fields. Strategy: embed the new `TimedItinerary` inside the existing response envelope (add fields, don't remove old ones). `RouteVisualization.tsx` continues working with `ordered_points`; the new `RoutePlannerWizard.tsx` reads `timed_itinerary`. Both consumers updated in Phase A Week 2.
