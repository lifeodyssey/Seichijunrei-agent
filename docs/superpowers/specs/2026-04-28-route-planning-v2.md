## Problem Statement

`plan_route` crashes with "Too many locations to route (max 50)" for any anime with
50+ spots (Your Name: 111, Haruhi: 76, Eupho: 134). The route optimizer uses a 50m
cluster threshold, producing ~80-100 clusters from 111 points, then hard-rejects >50.

Beyond the crash, the route model is fundamentally wrong. It assumes a single continuous
walking route through all spots. Real pilgrimage looks like:

1. Take transit to an area
2. Walk the spots in that area
3. Take transit to the next area

The route should reflect this cluster-walk-transit-cluster pattern, not
point-to-point walking across 25km of Tokyo.

## What Makes This Cool

The insight: use the LLM's world knowledge for the part that needs judgment (how to
split spots into travel areas, which station to use, what order makes sense for a day
trip) and algorithms for the part that's mechanical (optimal walking order within a
small cluster).

3-6 spots? Just connect them, no algorithm needed.
20+ spots across multiple neighborhoods? LLM splits them into walkable areas, algorithm
handles each area's walking route.

This is a genuinely novel approach to pilgrimage route planning. Google Maps can't do
"group these anime spots by the nearest train station and give me a walking route per
station." It requires domain knowledge (anime + Japan geography) that only an LLM has.

## Constraints

- No transit API in Phase 1 (LLM world knowledge instead)
- No station database in Phase 1 (LLM knows Japanese stations)
- Must not break existing plan_route for small point sets (<= ~10 points)
- PydanticAI agent architecture (tool-based)
- Univibe/GPT-5.5 as LLM (~20-40s per call, keep calls minimal)
- Frontend already has RoutePlannerWizard component

## Premises

1. **plan_route crashes on >50 clusters. Must fix.** Any popular anime breaks it.

2. **Route model = cluster-walk-transit-cluster.** Not a single walking path.
   Each cluster is walkable. Between clusters is transit.

3. **Geographic proximity > episode order.** At each area, walk all nearby spots
   regardless of episode. Episode info stays in UI but doesn't dictate route order.

4. **Cluster threshold is context-dependent.** Tokyo: 1-2km per walking area.
   Rural (Uji, Hida): 3-5km per walking area. A fixed threshold won't work.
   LLM decides the grouping based on its knowledge of the area.

5. **Small sets don't need optimization.** 3-10 points: existing nearest-neighbor
   is sufficient, no LLM splitting needed. LLM area-split activates only for >10
   points. 3-6 points can skip nearest-neighbor entirely and just connect in order.

## Approaches Considered

### Approach A: LLM splits areas + algorithm walks each area (recommended)

**How it works:**
1. If points <= 10: skip LLM, run existing nearest-neighbor directly
2. If points > 10: call LLM with a summary of points (name, lat/lng, episode),
   ask it to group them into walkable areas, name each area (e.g., "新宿駅周辺"),
   and suggest area visit order
3. For each area: run existing cluster_by_location (50m) + nearest_neighbor within
   the area's subset of points
4. Between areas: mark as "transit" leg (no distance calc, just label)
5. Return a multi-area itinerary

**LLM prompt (sketch):**
```
You have {N} anime pilgrimage spots. Group them into walkable areas (1-3km radius
each). For each area, name the nearest major train station and estimate walking time.
Output JSON: {"areas": [{"name": "新宿駅周辺", "station": "新宿駅",
"point_ids": ["id1", "id2", ...], "walk_minutes": 45}, ...],
"recommended_order": [0, 1, 2]}
```

**Effort:** M (new LLM tool call in plan_route + area-splitting logic)
**Risk:** Medium (LLM might group poorly; need fallback)
**Pros:**
- Adapts to any geography without hardcoded thresholds
- LLM knows stations, neighborhoods, walkability
- Graceful scaling: works for 10 spots or 200
**Cons:**
- Extra LLM call (~20s latency)
- LLM might hallucinate station names
- Non-deterministic grouping

### Approach B: Two-level fixed clustering

Two passes of cluster_by_location: first at 2km to create areas, then 50m within
each area for walking stops.

**Effort:** S-M
**Risk:** Low
**Pros:** Deterministic, no LLM call, fast
**Cons:** Fixed threshold doesn't adapt, doesn't know about stations, Tokyo vs rural
has different needs

### Approach C: Truncate + warn

When clusters > 50, take the 30 clusters with the most member points (highest
point count). Warn user.

**Effort:** S (10 lines)
**Risk:** Low
**Pros:** Ships today
**Cons:** Drops 80% of spots, walking-only route still wrong

## Recommended Approach

**Approach A + C as fallback.** Ship C immediately (prevent crash), then implement A
as the real solution. C stays as the fallback when LLM area-splitting fails.

### Implementation phases

**Phase 1 (immediate): Crash fix (Approach C)**
- In `optimize_route`, when clusters > 50, take top 30 by point density
- Add warning to result: "Selected 30 representative spots from {N} total"
- 10 lines of code, ships today

**Phase 2 (this iteration): LLM area splitting (Approach A)**
- New function: `split_into_areas(points, llm)` that asks LLM to group points
- Modify `plan_route` tool: if points > 10, call `split_into_areas` first
- Each area runs through existing `optimize_route` independently
- New response model: `AreaRoute` with name, station, points, walking route
- Frontend: show tabs or collapsible sections per area

**Phase 3 (future): Dedicated route planner agent**
- Independent PydanticAI agent specialized in route planning
- Own tools: station lookup, distance calculation, web search for transit info
- Main pilgrimage agent delegates to it

## Data Model

```python
class AreaPlan(BaseModel):
    """One walkable area in a multi-area route."""
    name: str                    # e.g., "新宿駅周辺"
    station: str = ""            # nearest station name
    point_ids: list[str]         # IDs of points in this area
    walk_minutes: int = 0        # estimated walking time within area
    order: int = 0               # visit order (0-based)

class TransitLeg(BaseModel):
    """Transit between two areas."""
    from_area: str
    to_area: str
    mode: str = "transit"        # "transit" | "walk" (if areas are close)
    note: str = ""               # e.g., "JR中央線 新宿→四ツ谷 (5分)"

class MultiAreaRoute(BaseModel):
    """Top-level route response for multi-area itineraries."""
    areas: list[AreaPlan]
    transit_legs: list[TransitLeg] = Field(default_factory=list)
    total_areas: int = 0
    total_points: int = 0
    source: str = "llm"          # "llm" | "truncated" (fallback)
    warning: str = ""            # non-empty if truncated
```

**Always use `MultiAreaRoute`** — even for small sets (<=10 points, single area).
One response shape for all route responses. Frontend only needs to handle one format.
(Eng review decision: option A, avoid two-shape branching.)

## Validation & Fallback

When LLM area-split is used:

| Failure condition | Action |
|---|---|
| LLM returns malformed JSON | Retry once with ModelRetry, then fall back to Approach C |
| point_ids don't cover all points | Log warning, assign orphans to nearest area by distance |
| Duplicate point_ids across areas | Deduplicate, keep in first area |
| Single area > 50 points | Apply Approach C (truncate to 30) within that area |
| LLM timeout (>30s) | Fall back to Approach C immediately |
| Empty areas array | Fall back to Approach C |

LLM call uses PydanticAI structured output (`output_type=AreaSplitResult`) to
force valid JSON. Temperature = 0 for deterministic grouping.

## UX During Route Planning

- SSE step events: `plan_route running` → `plan_route splitting_areas` →
  `plan_route walking_routes` → `plan_route done`
- Frontend shows: "Analyzing areas..." → "Planning walking route for 新宿駅周辺..."
  → "Done (3 areas, 45 spots)"
- 30s hard timeout on LLM area-split call. Fallback shows:
  "Selected 30 representative spots (full route requires more processing time)"

## Cost & Performance

- LLM area-split input: ~100 points × 30 tokens each = ~3K tokens input + ~500 tokens output
- One extra LLM call per route plan (only for >10 points)
- At current Univibe pricing, negligible (<$0.01 per route)
- Latency: +15-25s for the LLM call. Total route: ~30-50s for large sets.

## Open Questions

1. LLM area-split prompt: how many points to send? Full list or pre-filtered?
   (111 points with lat/lng is ~3KB, should be fine)
2. Should the LLM also suggest time allocation per area? Or fixed per-point dwell?
3. How to validate LLM's grouping? (All point_ids must be accounted for, no orphans)
4. Frontend: tabs per area vs scrollable timeline vs map with colored regions?

## Success Criteria

- plan_route completes successfully for any anime (even 200+ spots)
- Route output reflects real pilgrimage pattern (transit between areas, walk within)
- Small point sets (<=10) are not slower than today
- User can understand the route structure at a glance (which area first, which next)

## Eng Review (2026-04-28)

### Decisions

1. **Always `MultiAreaRoute`** — single response shape for all route sizes. Small
   sets = single area. No two-shape branching in frontend.
2. **`execute_selected_route` kept for now** — fast path (no LLM) for user-selected

### Sub-agent architecture

`plan_route` tool 内部 delegate 给 route planner sub-agent（Phase 2）。
PydanticAI 的 agent delegation = 在 tool 函数里调 `await sub_agent.run()`。
前端不感知 sub-agent，response 统一为 `MultiAreaRoute`。

```python
@pilgrimage_agent.tool
async def plan_route(ctx, *, origin, pacing, start_time):
    rows = search_data["rows"]
    if len(rows) <= 10:
        return optimize_route(rows, params, origin)  # 算法直走
    # delegate 给 route planner sub-agent
    route_agent = Agent(model, output_type=AreaSplitResult)
    split = await route_agent.run(
        f"Split these {len(rows)} spots into walkable areas...",
        usage=ctx.usage,  # 共享 token 计数
    )
    # 对每个 area 分别 optimize_route
    ...
```

### Eval strategy — two-layer evaluation

Best practice: outcome-level gate + trace-level diagnostics.
(Ref: [Evaluating LLM Agents in Multi-Step Workflows](https://www.codeant.ai/blogs/evaluate-llm-agentic-workflows),
[Anthropic: Demystifying Evals](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents))

**Layer 1: Gate (pass/fail, blocks merge)**

| Evaluator | Type | What it checks |
|-----------|------|---------------|
| IntentMatch | assertion | agent intent == "plan_route" |
| DataCompleteness | assertion | result has `areas`, all point_ids covered |

**Layer 2: Quality score (0-1, does not block)**

| Evaluator | Type | What it checks |
|-----------|------|---------------|
| AreaSplitQuality | score | >10 points → >1 area; each area has name+station; geographically distant points in separate areas (e.g., 飛騨 vs 新宿) |
| StepTrace | score | steps contain `route_planner` sub-agent call; helps debug "why did it fall back to truncate instead of LLM split" |

**Layer 3: LLM Judge (score, diagnostics)**

| Evaluator | Type | What it checks |
|-----------|------|---------------|
| RouteQualityJudge | LLMJudge | "Is this route plan practical for a real pilgrimage trip? Are areas walkable? Is the visit order logical?" |

**Existing 93 plan_route cases** get extended with:
- `expected_data_keys: ["areas"]` — triggers DataCompleteness check
- Case-specific LLMJudge rubrics for key anime (Your Name must split 飛騨/東京,
  Eupho should be 1-2 areas in Uji)

**New eval dataset** (`route_planner_eval.json`) for sub-agent unit testing:
- Input: list of point coordinates + names
- Expected: area grouping
- Tests `split_into_areas()` directly, not through main agent pipeline
- ~20 cases covering: single-area (Uji), multi-area (Your Name), cross-region
  (Slam Dunk), dense urban (Shinjuku), sparse rural (Hida)
