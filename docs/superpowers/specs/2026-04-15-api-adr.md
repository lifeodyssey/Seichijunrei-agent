```

   Output: { tool: "search_nearby", params: { location: "宇治駅", radius: 5000 } }

2. [Tool 1] search_nearby(location="宇治駅", radius=5000)
   ├─ Resolve location "宇治駅":
   │   ├─ Check KNOWN_LOCATIONS dict → miss
   │   ├─ [LLM call 2] Fuzzy match: "宇治駅" → (34.9058, 135.8074)
   │   └─ Coordinates: (34.9058, 135.8074)
   ├─ Retriever.execute(strategy=GEO)
   │   DB: SELECT p.*, b.title, b.title_cn,
   │       ST_Distance(
   │         COALESCE(p.location, ST_SetSRID(ST_MakePoint(p.longitude, p.latitude), 4326)::geography),
   │         ST_MakePoint(135.8074, 34.9058)::geography
   │       ) AS distance_m
   │       FROM points p JOIN bangumi b ON p.bangumi_id = b.id
   │       WHERE ST_DWithin(..., 5000)
   │       ORDER BY distance_m
   │       LIMIT 200
   │   → 89 rows (with distance_m on each)
   └─ Return: { rows: [...], row_count: 89, strategy: "geo" }

3. [LLM call 3] Planner: done. "89件見つかりました"
```

**LLM calls total: 3 (pipeline, includes location resolution)**

```
           b.title_cn, b.cover_url, b.city,
           COUNT(p.id) AS points_count
    FROM points p
    JOIN bangumi b ON p.bangumi_id = b.id
    WHERE ST_DWithin(
        COALESCE(p.location, ST_SetSRID(ST_MakePoint(p.longitude, p.latitude), 4326)::geography),
        ST_MakePoint($1, $2)::geography,
        $3
    )
    GROUP BY b.id
    ORDER BY points_count DESC
    LIMIT 10
```

**Existing code:** `BangumiRepository.get_bangumi_by_area(lat, lng, radius_m)` exists. Needs to add cover_url, title_cn, points_count to the query, plus HTTP wiring.

### 2.3 Adjust `PublicAPIRequest` — Coordinate-based origin

**Current `origin` field:** text string only ("宇治駅").
**Problem:** When user taps 📍 (browser geolocation), frontend has coordinates, not a place name.

**Change: add two optional fields:**
```python
class PublicAPIRequest(BaseModel):
    # ... existing fields ...
    origin: str | None = None              # text origin (existing)
    origin_lat: float | None = None        # NEW
    origin_lng: float | None = None        # NEW
```

**Behavior:**
- If `origin_lat`/`origin_lng` provided: skip location resolution, use directly as (lat, lng)
- If `origin` (text) provided: resolve via KNOWN_LOCATIONS → LLM → Geocoding (existing flow)
- If both provided: coordinates take precedence

**Impact on pipeline:**
```
plan_route handler:
  IF origin_lat AND origin_lng:
    origin_coords = (origin_lat, origin_lng)     ← NEW path, no LLM
  ELIF origin (text):
    origin_coords = resolve_location(origin)      ← existing path, may call LLM
  ELSE:
    origin_coords = None                          ← no origin, route from first point
```

**Saves 1 LLM call** when geolocation is available.

### 3.2 Decision: Enrichment strategy

**Option A: LLM batch enrichment (recommended)**
```
For each point where city IS NULL:
  1. Input: coordinates (lat, lng) + anime title
  2. LLM call: "Given coordinates 34.89, 135.80, what Japanese city/ward is this in?"
  3. Output: "宇治市"
  4. DB: UPDATE points SET city = $1 WHERE id = $2
  5. Cache: never re-enrich
```

Cost: ~$0.01 per 100 points (simple factual query). One-time per point. 2400 points = ~$0.24 total.

**Option B: Reverse geocoding API (alternative)**
```
Nominatim (free) or Google Geocoding:
  GET https://nominatim.openstreetmap.org/reverse?lat=34.89&lon=135.80&format=json
  → { "address": { "city": "宇治市", "state": "京都府" } }
```

Zero LLM cost but requires external API dependency and rate limiting.

**Decision: Option B for city, Option A for missing cn_name.**
- City: deterministic reverse geocode, no LLM needed
- CN name: LLM can translate JP location names to Chinese
