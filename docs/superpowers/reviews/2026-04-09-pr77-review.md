# PR #77 Review — streaming thought display with natural language

## Summary verdict

Request changes before merge.

The main ReAct streaming path is moving in the right direction, but the new step-event contract is not consistent end-to-end yet. I found two behavior/contract issues and one meaningful test gap that matter for this PR's stated goal.

## Findings

### Important

1. **Selected-points execution still reports failed steps as `done`**
   **Confidence:** 94/100
   **Files:** `/Users/lumimamini/Documents/Seichijunrei-agent/backend/agents/executor_agent.py:98`, `/Users/lumimamini/Documents/Seichijunrei-agent/backend/interfaces/public_api.py:123`
   **Why this matters:** `selected_point_ids` requests bypass the ReAct loop and go through `ExecutorAgent.execute()`. That path still emits `on_step(..., "done", ...)` before checking `step_result.success`, then breaks on failure. So the new `failed` status is never emitted for that runtime path, which means the frontend's red failure UI and the SSE contract now differ depending on how the route was produced. Users will see inconsistent behavior for the same failure concept.
   **Fix suggestion:** Emit `failed` when `step_result.success` is false, or centralize step-event emission so both ReAct and direct executor paths share one implementation.

2. **Frontend SSE typing is only partially updated for the new `failed` status**
   **Confidence:** 92/100
   **Files:** `/Users/lumimamini/Documents/Seichijunrei-agent/frontend/lib/types.ts:152`, `/Users/lumimamini/Documents/Seichijunrei-agent/frontend/lib/api.ts:129`, `/Users/lumimamini/Documents/Seichijunrei-agent/frontend/lib/api.ts:187`, `/Users/lumimamini/Documents/Seichijunrei-agent/frontend/tests/conversation-api.test.ts:199`
   **Why this matters:** `StepEvent.status` now includes `"failed"`, but the stream payload type and `sendMessageStream()` callback still only type status as `"running" | "done"`. At runtime the backend can now send `"failed"`, so TypeScript no longer matches the actual SSE contract. That weakens exhaustiveness checking right where this PR is changing behavior.
   **Fix suggestion:** Update `StreamEventPayload.status`, the `sendMessageStream()` callback signature, and the frontend stream tests to include `"failed"`.

3. **The tests were updated for the new callback arity, but not for the new SSE fields/failed-state contract**
   **Confidence:** 89/100
   **Files:** `/Users/lumimamini/Documents/Seichijunrei-agent/backend/tests/integration/test_sse_contract.py:89`, `/Users/lumimamini/Documents/Seichijunrei-agent/backend/tests/integration/test_sse_contract.py:209`, `/Users/lumimamini/Documents/Seichijunrei-agent/frontend/tests/conversation-api.test.ts:193`
   **Why this matters:** The PR threads `thought` and `observation` through the backend/frontend chain and adds a `failed` step state, but the updated tests only change mock invocations to 5 args. The integration test still asserts only `tool` and `status`, and the frontend stream tests still cover only `running`/`done`. That leaves the exact contract this PR introduces effectively unguarded.
   **Fix suggestion:** Add one backend SSE assertion for `thought`/`observation` presence and one frontend parser test that consumes a `failed` step event with thought/observation.

## What looks solid

- `/Users/lumimamini/Documents/Seichijunrei-agent/backend/agents/pipeline.py:199` correctly threads `thought` and `observation` from `ReactStepEvent` into `on_step`, so the core ReAct path now exposes the planner text end-to-end.
- `/Users/lumimamini/Documents/Seichijunrei-agent/backend/interfaces/fastapi_service.py:166` preserves backward compatibility nicely by giving `thought` and `observation` default empty strings in the SSE serializer callback.
- `/Users/lumimamini/Documents/Seichijunrei-agent/frontend/components/chat/ThinkingProcess.tsx:37` is a reasonable UI rewrite for streaming readability, and the failed-step styling is directionally good once the contract issues above are fixed.

## Recommended follow-ups

- Unify step-event emission so ReAct and direct executor paths cannot drift on status semantics.
- Update all frontend stream types to the real contract: `running | done | failed` plus optional `thought`/`observation`.
- Extend SSE tests to assert the new fields explicitly, not just the callback shape.
